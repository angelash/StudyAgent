# 小学学习闭环系统需求分析

## 1. 文档目的

本文基于当前工程已有文档与教材索引，输出一套面向小学阶段的学习闭环系统需求分析，目标是为后续 Web 平台实现提供统一的产品边界、功能范围和优先级判断依据。

本需求分析主要依据两类输入：

1. 当前工程文档 `external-2026-孩子为什么能长时间打游戏却难以专注写作业.md`
2. 教材索引仓库 `E:\ChinaTextbook` 中的小学阶段教材目录结构

其中，现有研究材料给出了四个关键设计原则：

1. 目标可视化
2. 反馈即时化
3. 难度个性化
4. 失败安全化

教材索引则提供了稳定的内容骨架，至少已经具备以下基础层级：

1. 学段：小学
2. 学科：语文、数学、英语、科学、道德与法治等
3. 版本：人教版、统编版、北师大版等
4. 册次：一年级上/下册到六年级上/下册
5. 资源形态：PDF 教材文件

这意味着系统可以围绕“教材目录作为索引，知识点作为能力单元，任务系统作为训练载体，评估与反馈作为闭环驱动”来建设。

---

## 2. 产品愿景

构建一个以教材知识为主线、以儿童学习体验为核心、以家长和教师协同为辅助的 Web 学习平台，让小学阶段学生在“评估、训练、正面反馈、跟踪、再评估”的循环中持续进步，而不是被动完成作业。

系统不追求“把学习做成游戏”，而是借鉴优秀游戏系统的结构优势：

1. 让学生始终知道“下一步做什么”
2. 让学生快速知道“刚刚有没有做对”
3. 让任务尽量落在“踮脚能够到”的难度区间
4. 让错误成为可恢复的学习信号，而不是羞耻来源

---

## 3. 业务目标

### 3.1 学生目标

1. 降低开始学习的启动成本
2. 提高单次学习的专注时长与完成率
3. 提升对当前教材知识点的掌握度
4. 建立稳定、正向的“我能学会”体验
5. 形成可持续的复习与巩固习惯

### 3.2 家长目标

1. 看得见孩子当前学到哪里
2. 看得见孩子真实卡点，而不是只看总分
3. 获得可执行的陪伴建议，而不是泛泛焦虑
4. 在鼓励与纠偏时有系统支持

### 3.3 平台目标

1. 建立教材到知识点到任务到评估结果的完整链路
2. 支持多学科、多版本扩展
3. 形成可运营、可持续维护的内容与数据资产
4. 先服务小学阶段，再逐步扩展到更高学段

---

## 4. 范围定义

### 4.1 本期目标范围

面向小学阶段，设计一套完整闭环系统，包含：

1. 教材索引体系
2. 学生画像与能力模型
3. 诊断评估体系
4. 日常训练体系
5. 正面反馈机制
6. 学习追踪体系
7. 家长协同机制
8. 管理后台与内容运营体系
9. 基于 Web 的学生端、家长端、运营端产品形态

### 4.2 首期实施建议范围

系统设计支持全小学学科，但首期实现建议按以下顺序推进：

1. 数学
2. 语文
3. 英语
4. 科学与道德与法治

原因如下：

1. 数学知识结构最清晰，最适合先验证评估与训练闭环
2. 语文与英语适合在已有闭环之上补充语言类训练模块
3. 科学、道法等更适合在平台能力成熟后进入

### 4.3 非目标范围

首期不将以下内容作为核心交付目标：

1. 完整替代学校课堂教学
2. 公开排行榜驱动的强竞争机制
3. 高沉浸娱乐化设计
4. 复杂社交社区
5. 以考试排名为唯一导向的学习模式

## 5. 目标用户与角色

### 5.1 核心用户

#### 学生

按认知与交互能力划分三个层次：

1. 低年级：1-2 年级
2. 中年级：3-4 年级
3. 高年级：5-6 年级

#### 家长

家长负责：

1. 初始建档
2. 学习计划约束
3. 陪伴反馈
4. 学习结果查看

### 5.2 次级用户

#### 教师/辅导者

用于查看班级或个体学习进展，布置补充任务，分析共性问题。

#### 内容运营/教研人员

负责教材结构维护、知识点标注、题目维护、反馈模板维护和策略优化。

#### 系统管理员

负责用户权限、内容发布、版本管理、日志与审核。

## 6. 核心问题定义

结合当前工程文档，传统学习场景存在以下典型问题：

1. 任务目标模糊，孩子不知道从哪开始
2. 反馈延迟，无法及时感知自己的努力是否有效
3. 任务难度不匹配，过易无聊，过难挫败
4. 错误代价过高，导致回避挑战
5. 家长只能看到结果，无法看到过程
6. 教材与习题、反馈、计划之间没有形成数据闭环

系统需要正面回答以下问题：

1. 学生此刻最应该学什么
2. 这一题为什么给他做
3. 做完之后系统如何即时反馈
4. 做错之后如何降台阶而不是直接判定失败
5. 如何让家长看到“进步轨迹”而不是“单次成败”
6. 如何从教材资源逐步构建可用的知识图谱

---

## 7. 闭环定义

本系统的学习闭环定义如下：

1. 建档：确定年级、教材版本、学习目标、时间约束
2. 评估：通过诊断测试识别当前掌握水平和薄弱点
3. 规划：生成本周计划和今日任务
4. 训练：进行知识点训练、巩固、错题回练和复习
5. 反馈：对过程和结果做即时、具体、正向反馈
6. 跟踪：沉淀掌握度、错误类型、完成习惯和情绪状态
7. 调整：根据数据重新调整难度、节奏和任务结构
8. 再评估：在单元、周、阶段节点再次校准学习路径

系统必须保证以上 8 个环节可以串联，不能只有“刷题”或只有“报告”。

## 8. 产品原则

### 8.1 儿童优先

所有设计优先服务孩子的认知负担、操作习惯和情绪体验，而不是服务后台管理便利。

### 8.2 小步快跑

每次任务必须足够小，让学生容易开始，也容易完成。

### 8.3 过程反馈优先于结果反馈

系统不仅告诉学生“对或错”，更要告诉学生“哪一步做对了、哪一步值得保留、下一步怎么调整”。

### 8.4 自我比较优先于同伴比较

系统鼓励“比昨天更好”，避免用公开排名放大焦虑。

### 8.5 内容可溯源

学习任务必须能追溯到教材版本、册次、单元、课时或知识点，保证体系化。

### 8.6 失败安全

做错题后要优先进入提示、拆分、示范、重试，而不是直接结束。

## 9. 关键场景

### 9.1 首次使用

1. 家长创建孩子档案
2. 选择年级、教材版本、在学学科
3. 设定每日学习时长和学习目标
4. 学生完成轻量诊断评估
5. 系统生成初始能力画像和第一周计划

### 9.2 每日学习

1. 学生进入首页，看到“今日任务”
2. 系统把任务拆成 10-20 分钟的学习单元
3. 学生在训练中得到即时反馈
4. 学习完成后获得阶段总结和正向鼓励
5. 家长可在日视图中看到完成情况与建议

### 9.3 单元巩固

1. 学生学完某教材单元后进入单元评估
2. 系统输出知识点掌握热力图
3. 自动生成巩固包和错因分析
4. 下周计划优先覆盖薄弱项

### 9.4 学习卡住

1. 学生连续做错或停留过久
2. 系统识别为“困难状态”
3. 自动降阶、给提示、切换示例题
4. 若仍卡住，则生成家长陪伴建议或老师干预建议

## 10. 功能需求

### 10.1 教材索引与内容组织

系统需要基于 `E:\ChinaTextbook` 构建内容索引层，至少支持以下字段：

1. 学段
2. 学科
3. 版本
4. 年级
5. 上下册
6. 教材文件路径
7. 单元
8. 课时/章节
9. 知识点
10. 对应训练资源

具体要求如下：

1. 能扫描教材目录，识别小学阶段各学科版本和册次
2. 支持人工或半自动补录教材目录结构
3. 支持知识点挂靠到教材章节
4. 支持同一知识点关联多个题目、讲解、复习包
5. 支持不同版本教材映射到统一能力标签

### 10.2 学生档案与学习画像

每个学生需要维护一套可持续更新的学习画像，至少包括：

1. 基本信息：年级、年龄、地区、教材版本
2. 学科配置：当前学习学科、优先级
3. 学习偏好：题型偏好、时长偏好、学习时段
4. 学习状态：专注度、完成度、坚持天数
5. 能力画像：知识点掌握度、能力维度、错误类型
6. 风险标签：畏难、粗心、连续低完成、基础缺失

### 10.3 评估系统

评估系统分为五类：

1. 入门诊断评估
2. 日常微评估
3. 单元评估
4. 阶段评估
5. 错题回测评估

评估系统需要满足以下要求：

1. 评估结果能定位到知识点层级
2. 评估不能只给总分，要输出掌握区间和风险点
3. 评估要能驱动后续任务推荐
4. 评估过程要尽量短，适合小学用户完成
5. 评估支持分层难度和自适应出题

#### 10.3.1 入门诊断评估

目标：

1. 确定起始水平
2. 找到明显前置缺口
3. 给出初始任务分层

#### 10.3.2 微评估

目标：

1. 在每天学习过程中快速校准理解状态
2. 检测是否可以进入下一难度
3. 避免长时间在错误难度层停留

#### 10.3.3 错题回测

目标：

1. 判断是偶发错误还是稳定不会
2. 识别是否真正完成迁移
3. 决定是否再次纳入复习

### 10.4 训练系统

训练系统是闭环中的核心执行层，需要提供至少五种训练模式：

1. 今日任务
2. 知识点闯关
3. 错题重练
4. 间隔复习
5. 单元巩固包

训练系统必须满足以下要求：

1. 训练目标清晰，学生进入后能立刻开始
2. 每个任务控制在儿童可接受时长内
3. 训练中支持提示、拆步、示范和重试
4. 训练结束后有明确结论和下一步建议
5. 训练记录会实时写入学生画像

#### 10.4.1 年级差异要求

#### 1-2 年级

1. 以图形、语音、少字界面为主
2. 单次任务建议 8-12 分钟
3. 更强调即时鼓励和演示式引导

#### 3-4 年级

1. 可接受更多文字指令
2. 单次任务建议 12-18 分钟
3. 引入简单自我检查和错因选择

#### 5-6 年级

1. 支持更长链路任务
2. 单次任务建议 15-25 分钟
3. 引入更明确的计划与复盘机制

### 10.5 正面反馈机制

正面反馈不是简单发积分，而是系统核心能力。

反馈应分为四层：

1. 即时反馈：每题、每步、每次点击后的结果确认
2. 阶段反馈：完成一个知识点或一个学习单元后的总结
3. 周期反馈：周报、单元报告、成长趋势
4. 协同反馈：家长或教师收到的鼓励建议

反馈机制要求：

1. 尽量具体，不说空泛的“你真棒”
2. 优先肯定过程中的正确策略
3. 做错时也要保护尝试意愿
4. 避免高压或羞辱型话术
5. 避免高度依赖外部奖赏，逐步转向能力感与成长感

#### 10.5.1 反馈触发规则

至少应覆盖以下触发场景：

1. 第一次完成某知识点
2. 连续答对
3. 从连续错误中恢复
4. 今日任务完成
5. 连续学习达成
6. 比前一次明显进步
7. 难点被攻克

#### 10.5.2 反馈表现形式

1. 文本鼓励
2. 进度条
3. 成长树/勋章/拼图等低成瘾视觉奖励
4. 学习卡片
5. 家长端的鼓励建议卡

### 10.6 跟踪系统

跟踪系统必须覆盖“学了什么、学得怎样、卡在哪里、是否稳定、是否持续”五个层面。

核心跟踪维度包括：

1. 教材进度
2. 知识点掌握度
3. 题目正确率
4. 错误类型
5. 学习时长
6. 完成率
7. 复习命中率
8. 连续学习天数
9. 情绪与挫败信号
10. 家长参与度

#### 10.6.1 学生端可见数据

1. 今日完成情况
2. 本周成长点
3. 当前正在攻克的能力点
4. 下一步建议

#### 10.6.2 家长端可见数据

1. 学习总览
2. 学科进度
3. 掌握热力图
4. 最近卡点
5. 建议陪伴动作

#### 10.6.3 教师/运营端可见数据

1. 群体分布
2. 高风险知识点
3. 训练完成情况
4. 题目质量反馈

### 10.7 计划与推荐

系统应能根据评估结果、教材进度和学生状态，自动生成学习计划。

推荐逻辑至少考虑：

1. 当前教材所在位置
2. 前置知识是否稳固
3. 最近错误频率
4. 学习时长预算
5. 当前学习信心和专注状态
6. 是否到达复习时间窗口

系统输出至少包括：

1. 今日任务
2. 本周目标
3. 优先攻克知识点
4. 可选挑战任务
5. 必做复习任务

### 10.8 家长协同

家长端不是监工端，而是陪伴端。

需要提供：

1. 孩子建档与教材选择
2. 学习时长与节奏设置
3. 每日/每周报告查看
4. 鼓励话术建议
5. 卡点干预建议
6. 成就共同庆祝提醒

不建议提供：

1. 公开排名对比
2. 过度惩罚配置
3. 用分数作为唯一反馈入口

### 10.9 内容运营与教研后台

后台至少支持：

1. 教材版本维护
2. 单元与章节结构维护
3. 知识点树维护
4. 题目和讲解维护
5. 反馈模板维护
6. 规则策略维护
7. 用户学习日志查看
8. 数据看板

## 11. 业务规则

### 11.1 难度调节规则

1. 连续正确且耗时稳定，可适当提高难度
2. 连续错误时优先降阶，而不是继续堆相似难题
3. 超出当前能力范围的题应推迟到合适时机
4. 训练目标难度应尽量处于“可挑战可完成”区间

### 11.2 失败安全规则

1. 学生答错后优先提供提示和重试机会
2. 连续失败时自动切换到更基础任务
3. 重要页面避免出现否定性结论
4. 报告中不出现羞辱性标签

### 11.3 反馈规则

1. 正确反馈要指出“具体做对了什么”
2. 错误反馈要指出“下一步怎么做”
3. 对低年级用户，反馈字数和复杂度要受控
4. 积分或勋章只能作为辅助，不可替代有效教学反馈

### 11.4 复习规则

1. 新学知识点必须进入复习计划
2. 错题必须进入回测池
3. 对“看似会、过几天忘”的知识点，要加强间隔复习

## 13. 非功能需求

### 13.1 可用性

1. 面向小学生，操作路径必须短
2. 首页必须一眼看见“现在做什么”
3. 页面信息密度不能过高

### 13.2 性能

1. 主要页面应快速打开
2. 题目提交与反馈要接近实时
3. 报告计算应支持异步生成与缓存

### 13.3 可扩展性

1. 支持扩展更多学科与版本
2. 支持从小学扩展到初中
3. 支持规则引擎逐步升级为更智能的推荐引擎

### 13.4 安全与隐私

1. 儿童数据必须受保护
2. 家长授权关系必须清晰
3. 日志与报告访问要有权限控制
4. 内容发布与审核要有留痕

### 13.5 可运营性

1. 内容运营人员可维护教材结构
2. 教研人员可维护知识点与题目
3. 数据团队可查看关键指标

---

## 14. 成功指标

首期建议关注以下指标：

1. 首次建档完成率
2. 首次评估完成率
3. 今日任务完成率
4. 单次学习中断率
5. 学习后次日回访率
6. 知识点掌握度提升率
7. 错题二次正确率
8. 周留存
9. 家长查看报告率
10. 学生连续学习达成率

同时建议引入体验型指标：

1. 学生是否清楚下一步做什么
2. 学生是否愿意继续下一轮学习
3. 家长是否理解孩子卡点

## 15. MVP 建议

### 15.1 MVP 目标

验证以下三个核心假设：

1. 基于教材索引的学习路径是可落地的
2. 即时反馈和难度调节能显著提升完成率
3. 家长端看到过程型数据后，陪伴质量会提高

### 15.2 MVP 功能范围

建议 MVP 聚焦：

1. 小学数学
2. 人教版或一个主流版本
3. 3 个年级样本
4. 评估、训练、反馈、跟踪四大闭环核心能力
5. 学生端、家长端、运营端最小可用版本

### 15.3 MVP 暂不做

1. 全学科全版本一次性上线
2. 大规模 AI 自动出题
3. 复杂社交互动
4. 过于花哨的游戏外壳

---

## 16. 风险与前置假设

### 16.1 风险

1. 教材 PDF 只有文件层索引，章节和知识点仍需补结构化
2. 多版本教材之间的知识点映射需要教研支持
3. 语文和英语的能力评估比数学复杂
4. 如果只做积分化反馈，可能会偏离真正学习闭环

### 16.2 前置假设

1. 平台允许通过人工或半自动方式建设知识点树
2. 首期由教研人员完成重点学科的内容整理
3. Web 平台作为统一入口，未来可扩展移动端

---

## 17. 需求结论

本系统不是单纯的题库，也不是单纯的教材阅读器，而是一个围绕教材索引构建的“小学学习操作系统”。

其最核心的能力闭环是：

1. 用教材建立学习主线
2. 用评估识别真实起点
3. 用训练推进小步进阶
4. 用正面反馈保护学习意愿
5. 用持续跟踪驱动计划调整

只要这个闭环真正跑起来，系统才能同时解决“学什么、怎么学、是否学会、如何继续学”这四类问题。
